# 统一输入来源选择设计

日期：2026-07-15

## 目标

每个转换任务的输入区域只保留一个选择入口。用户可以把单曲文件或文件夹拖入输入区域，也可以点击同一个按钮选择；程序把两种结果都作为输入来源交给既有预检和转换逻辑，不再要求用户区分“选择文件夹”和“选择单曲”。

## 用户体验

- 输入按钮文案改为“选择来源”（英文为 “Choose source”）。
- 删除“选择单曲”按钮，输入区域保留清空按钮。
- 拖拽行为保持不变：拖入路径后直接保存该路径，后端根据路径是文件还是文件夹进行校验。
- 点击“选择来源”时使用一个同时允许选择文件和文件夹的系统选择窗口；取消选择不改变当前来源。
- 输出目录仍使用单独的文件夹选择窗口。
- 不支持的文件类型仍由现有后端校验和错误提示处理，不在选择窗口中提前改变转换规则。

## 结构与数据流

前端将两个输入选择服务合并为 `pickSource(slotIndex)`。点击输入按钮时调用它，返回路径后继续复用现有的 `selectSourceDirectory(slotIndex, path)` 命令，因此任务状态、路径保存、预检和转换代码无需分叉。

桌面端新增一个 `pick_source_path` 命令：

- macOS 使用原生 `NSOpenPanel`，同时开启选择文件与选择文件夹，单选并返回所选路径。
- Windows/Linux 保留可用的桌面回退路径；回退实现仍返回单一路径，前端不会重新引入第二个可见按钮。
- 取消选择返回 `null`，系统错误转成用户可见的选择失败错误。

拖拽路径不经过选择窗口，继续由现有的 Tauri/browser drop 处理器直接交给 `selectSourceDirectory`。

## 测试与验收

前端测试覆盖：

- 渲染结果不再包含 `pick-source-file` 或“选择单曲”。
- 点击统一输入按钮会调用 `pickSource`，并把返回路径保存为任务来源。
- 单曲拖拽仍保存单曲路径。
- 文件夹选择仍保存文件夹路径。

桌面端测试覆盖选择命令的取消/返回路径辅助逻辑；现有单曲来源校验测试继续保留。完成前运行前端全量测试、Vite 构建、Rust 全量测试、Clippy，并重新构建 Debug app 手动确认两个任务卡片只显示一个输入选择按钮。

## 范围边界

任务卡底部的详情/日志区域也一并精简：

- 删除左侧日志列表图标。
- 删除右侧“查看歌曲详情/收起歌曲详情”按钮。
- 删除按钮展开的日志内容区域。
- 保留当前状态/进度文字和水平进度条。
- 后端日志数据继续保留给错误处理和历史记录使用，但不再在任务卡中展示。
